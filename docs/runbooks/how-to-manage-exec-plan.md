# Runbook: как вести execution plan

## Цель
- Сделать выполнение нетривиальных задач предсказуемым и проверяемым.
- Зафиксировать единый процесс для людей и агентов.

## Когда создавать план
- Перед любой нетривиальной задачей (3+ осмысленных шага, рискованное изменение поведения, кросс-модульная правка).
- Перед изменениями публичных контрактов, архитектурных границ или CI/guardrails.
- Можно не создавать только для тривиальных однофайловых правок без изменения поведения.

## Где создавать план
- Новый план: `docs/exec-plans/active/<task-id>.md`.
- Шаблон: `docs/exec-plans/TEMPLATE.md`.
- Формат task id: `<topic>-itNN-<short-name>`.

## Чеклист качества плана
- В контексте понятно, зачем нужна задача и что входит/не входит в scope.
- Каждый шаг указывает конкретные файлы/модули.
- В рисках есть реальные точки отказа и путь отката.
- В разделе верификации есть точные команды и проверяемые критерии.

## Процесс выполнения
1. Создать план по шаблону и заполнить все разделы.
2. Предложить scope изменений (файлы/директории) и получить явное подтверждение.
3. Выполнять работу небольшими батчами и обновлять чеклист по ходу.
4. После каждого батча запускать проверки из раздела верификации.
5. При блокере зафиксировать его в плане и эскалировать с вариантами решения.

## Правила обновления статусов
- Отмечать пункт `[x]` только при наличии доказательства (вывод команды, тест, ручная проверка).
- Не закрывать чеклист «пакетом» без фактической проверки каждого пункта.
- В рамках одного батча держать не более одного активного пункта.
- При изменении scope сначала обновлять план, затем код.

## Протокол завершения
1. Убедиться, что все пункты плана закрыты.
2. Выполнить финальную верификацию из плана.
3. Обновить связанные docs/ADR, если затронуто поведение или архитектура.
4. Перенести план в `docs/exec-plans/completed/`.
5. В финальном отчёте дать путь к завершённому плану и ключевые артефакты.

## Быстрый шаблон эскалации блокера
- **Что заблокировано:** короткое описание симптома.
- **Что уже пробовали:** конкретные попытки и результат.
- **Влияние:** что невозможно доставить прямо сейчас.
- **Варианты:** 2-3 безопасных пути с рекомендацией.

## Минимальная верификация для docs-only задач
- План существует и соответствует структуре шаблона.
- Все упомянутые пути в docs валидны.
- Отмеченные пункты чеклиста соответствуют фактическому состоянию репозитория.
